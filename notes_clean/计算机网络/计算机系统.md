## 加密体系基础

> **一句话理解：** 对称加密快但密钥传输不安全，非对称加密安全但慢，HTTPS 结合两者优点。

**核心结论（可背）：**
| 类型 | 代表算法 | 密钥特点 | 速度 | 场景 |
|---|---|---|---|---|
| 对称加密 | AES、DES、ChaCha20 | 加解密同一密钥 | 快 | 大量数据加密（TLS 通信阶段） |
| 非对称加密 | RSA、ECDHE | 公钥加密，私钥解密 | 慢 | 密钥交换、数字签名（TLS 握手阶段） |
| 摘要算法 | MD5、SHA-256 | 单向不可逆 | 快 | 数据完整性校验（数字指纹） |

**为什么 HTTPS 不全程用非对称加密？**
非对称加密计算复杂度高，加密速度慢，不适合加密大量数据；对称加密速度是非对称的数倍，适合持续通信。

## HTTPS 三大安全保障机制

> **一句话理解：** 保密性靠加密，完整性靠摘要，身份验证靠证书。

**核心结论（可背）：**
```
保密性（机密性）：
  混合加密 = 非对称（握手）+ 对称（通信）
  任何第三方截获报文，无法解密

完整性：
  摘要算法（如 SHA-256）对数据生成「数字指纹」
  接收方重新计算指纹，与附带指纹对比
  任何篡改都会导致指纹不匹配

身份验证：
  CA 数字证书：权威机构对服务器公钥的背书
  客户端验证证书有效性 → 确认服务器身份真实
```

## HTTP 版本演进

> **一句话理解：** 每一版都在解决上一版的性能或安全缺陷。

**核心结论（可背）：**
| 版本 | 核心特性 | 主要问题 |
|---|---|---|
| HTTP/1.0 | 短连接，每次请求新建 TCP | 连接开销大 |
| HTTP/1.1 | 持久连接（keep-alive），管道化 | 队头阻塞（HOL Blocking） |
| HTTP/2 | 多路复用（同一 TCP 多流并发），头部压缩 | TCP 层队头阻塞 |
| HTTP/3 | 基于 QUIC（UDP），内置 TLS 1.3 | 彻底解决队头阻塞 |

**队头阻塞（HOL Blocking）：**
```
HTTP/1.1 管道化：请求必须按顺序返回
  → 前一个响应慢，后面全部阻塞

HTTP/2 多路复用：TCP 层仍是顺序字节流
  → 一个 TCP 包丢失，整个连接所有流阻塞

HTTP/3（QUIC）：每个流独立，一个流丢包不影响其他流
  → 真正解决队头阻塞
```

## TLS 1.2 vs TLS 1.3

**核心结论（可背）：**
| 维度 | TLS 1.2 | TLS 1.3 |
|---|---|---|
| 握手 RTT | 2-RTT | 1-RTT（0-RTT 重连） |
| 密钥交换 | RSA（不具备前向安全） | 强制 ECDHE（前向安全） |
| 支持的套件 | 多（含不安全的） | 精简，只保留安全套件 |

**前向安全性：**
即使服务器私钥泄露，也无法解密历史通信记录。TLS 1.3 的 ECDHE 每次握手生成新的临时密钥，实现前向安全。

## 面试高频考点汇总

| 考点 | 核心答案 |
|---|---|
| 对称 vs 非对称加密区别？ | 密钥数量（一个 vs 两个）；速度（快 vs 慢）；用途（通信 vs 握手） |
| HTTPS 三大保障？ | 保密性（混合加密）、完整性（摘要算法）、身份验证（CA 证书） |
| HTTP 各版本核心改进？ | 1.1 持久连接；2 多路复用；3 QUIC 消除队头阻塞 |
| 什么是队头阻塞？ | 前一个请求/数据包阻塞导致后续全部等待 |
| TLS 1.3 比 1.2 好在哪？ | 1-RTT 握手、强制前向安全（ECDHE）、精简套件 |
| 摘要算法的作用？ | 生成数据指纹，验证完整性，防篡改 |
