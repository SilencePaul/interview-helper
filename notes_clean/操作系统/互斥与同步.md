# 互斥与同步

---

## 速览

- 互斥（Mutual Exclusion）= 保证同一时刻只有一个进程访问临界区。
- 同步（Synchronization）= 协调多个进程按特定顺序执行（有依赖关系的协作）。
- 死锁四个必要条件：互斥、请求与保持、不剥夺、循环等待——破坏任一即可预防。
- PV 操作 = 信号量的原子增减，P 申请资源（可能阻塞），V 释放资源（可能唤醒）。
- 三大经典问题：生产者-消费者、哲学家进餐、读者-写者。
- 银行家算法 = 死锁避免算法，每次分配前检查是否存在安全序列。

---

## 互斥 vs 同步

> **一句话理解：** 互斥关注"资源不能同时访问"，同步关注"操作按顺序发生"。

**核心结论（可背）：**
| 维度 | 互斥（Mutual Exclusion） | 同步（Synchronization） |
|---|---|---|
| 目标 | 防止数据竞争，保护临界区 | 协调执行顺序，满足依赖关系 |
| 典型场景 | 多个进程写同一文件 | 生产者放产品后消费者才能取 |
| 实现工具 | mutex、二进制信号量、自旋锁 | 计数信号量、条件变量、管程 |

**实现方式对比：**
| 实现方式 | 用于互斥 | 用于同步 | 是否忙等 |
|---|---|---|---|
| mutex（互斥锁） | ✅ | ❌ | ❌（阻塞） |
| semaphore（信号量） | ✅ | ✅ | ❌（阻塞） |
| spinlock（自旋锁） | ✅ | ❌ | ✅（忙等） |
| condition variable（条件变量） | ❌ | ✅ | ❌（阻塞） |
| 原子指令（CAS/TAS） | ✅ | ❌ | ✅（忙等） |

## 死锁

> **一句话理解：** 死锁 = 进程循环等待资源，形成环路，所有人都无法继续。

**核心结论（可背）：**

**死锁四个必要条件（背！）：**
```
① 互斥条件：资源同一时刻只能被一个进程使用
② 请求与保持：进程持有资源的同时请求新资源，且不释放已有资源
③ 不剥夺条件：进程已获得的资源，未使用完不能被强制剥夺
④ 循环等待：进程资源等待链形成环路
  → 破坏任意一个条件即可预防死锁
```

**死锁示意：**
```
进程 A 持有 R1，等待 R2
进程 B 持有 R2，等待 R1
→ 形成循环等待 → 死锁
```

**三种处理策略：**
| 策略 | 方式 | 优点 | 缺点 |
|---|---|---|---|
| 死锁预防 | 破坏四个必要条件之一 | 简单直接 | 资源利用率低 |
| 死锁避免 | 银行家算法动态检测安全状态 | 灵活，利用率高 | 实现复杂，开销大 |
| 死锁检测与恢复 | 允许死锁，定期检测，检测后撤销进程/抢占资源 | 最大利用率 | 存在风险，恢复复杂 |

**预防死锁的具体方法：**
```
破坏互斥条件：基本不可行（很多资源本身不可共享，如打印机）
破坏请求与保持：一次性申请所有资源（代价高，资源浪费）
破坏不剥夺条件：请求失败时主动释放已占资源（实现复杂）
破坏循环等待：对资源统一编号，进程必须按编号递增顺序申请资源
```

**易错点：**
- ❌ 死锁一定要破坏全部四个条件 → 只需破坏一个即可预防。
- ❌ 死锁避免 = 死锁预防 → 避免是动态检测（银行家算法），预防是静态约束。

## 银行家算法

> **一句话理解：** 每次分配资源前模拟执行，只有系统仍处于"安全状态"才真正分配。

**核心结论（可背）：**

**四个数据结构：**
| 结构 | 含义 |
|---|---|
| Available[m] | 系统当前各类资源可用数量 |
| Max[n][m] | 每个进程对各类资源的最大需求量（预先声明） |
| Allocation[n][m] | 当前已分配给每个进程的各类资源数量 |
| Need[n][m] | 每个进程还需要的资源量 = Max - Allocation |

**算法流程：**
```
进程 Pi 申请资源 Request[i]：
① Request[i] ≤ Need[i]？否则报错（超出最大需求）
② Request[i] ≤ Available？否则等待（资源不足）
③ 试分配：Available -= Request[i]，Allocation[i] += Request[i]，Need[i] -= Request[i]
④ 安全性检查：能否找到安全序列（所有进程按顺序都能完成）？
   - 是 → 正式分配
   - 否 → 回滚试分配，Pi 等待
```

**安全性检查（安全序列）：**
```
Work = Available（初始）
Finish[i] = FALSE（初始）

循环找满足条件的进程：Finish[i] == FALSE && Need[i] ≤ Work
  找到 → Work += Allocation[i]，Finish[i] = TRUE，继续循环
  找不到 → 停止

若所有 Finish[i] == TRUE → 安全状态 ✅
否则 → 不安全状态，拒绝此次分配 ❌
```

**优缺点：**
```
优点：严格避免死锁，资源利用率高于死锁预防
缺点：
  ① 需要预先声明最大需求（现实中难以预知）
  ② 每次请求需遍历所有进程：O(n²) 复杂度
  ③ 不适用于不可抢占资源（打印机、独占设备）
```

## PV 操作与信号量

> **一句话理解：** P 操作申请资源（减 1，可能阻塞），V 操作释放资源（加 1，可能唤醒）。

**核心结论（可背）：**
```
信号量 S 的含义：
  S > 0：S 表示当前可用资源数量
  S = 0：资源刚好用完，无进程等待
  S < 0：|S| 表示正在等待该资源的进程数量

P 操作（wait / 申请）：S--，若 S < 0，当前进程进入等待队列阻塞
V 操作（signal / 释放）：S++，若 S ≤ 0，从等待队列唤醒一个进程

关键特性：P/V 操作是原子操作（通过禁用中断或硬件指令保证）
```

**二进制信号量 vs 计数信号量：**
| 类型 | 取值范围 | 用途 |
|---|---|---|
| 二进制信号量（互斥信号量） | 0 或 1 | 临界区互斥保护，等价于 mutex |
| 计数信号量（同步信号量） | 任意整数 | 管理多个同类资源（如缓冲区空槽数量） |

## 三大经典同步问题

> **一句话理解：** 三大问题都用信号量解决，关键是找出互斥量和同步量。

**① 生产者-消费者问题：**
```
有界缓冲区，大小为 n
信号量设置：
  mutex = 1    → 互斥访问缓冲区
  empty = n    → 空槽数量（生产者的同步量）
  full  = 0    → 已有产品数（消费者的同步量）

生产者：P(empty) → P(mutex) → 放入产品 → V(mutex) → V(full)
消费者：P(full)  → P(mutex) → 取出产品 → V(mutex) → V(empty)

注意：P(mutex) 必须在 P(empty)/P(full) 之后，否则可能死锁
```

**② 哲学家进餐问题：**
```
5 个哲学家围桌，5 根筷子，每人需要左右两根才能吃
死锁场景：5 人同时拿起左边筷子 → 循环等待

解决方案（破坏循环等待）：
① 最多允许 4 个哲学家同时进餐（破坏循环等待）
② 只有左右两根都能拿到才允许进餐（破坏请求与保持）
③ 奇数号先拿左边，偶数号先拿右边（破坏循环等待）
```

**③ 读者-写者问题：**
```
允许多个读者同时读，但写者独占
三种优先级：
  读者优先：只要有读者在读，后来读者就能进入；写者可能饥饿
  写者优先：有写者等待时，禁止新读者进入；读者可能饥饿
  读写公平：按请求到达顺序分配权利（FIFO）
```

## 各类锁

> **一句话理解：** 互斥锁阻塞等待，自旋锁忙等待，读写锁允许并发读，乐观锁减少加锁。

**核心结论（可背）：**
| 锁类型 | 特点 | 适用场景 |
|---|---|---|
| 互斥锁（Mutex） | 独占访问，等待时进程阻塞（让出 CPU） | 临界区访问时间较长 |
| 自旋锁（Spinlock） | 忙等待，不让出 CPU，循环检查 | 临界区极短，锁持有时间短 |
| 读写锁（RWLock） | 多读者并发，写者独占 | 读多写少场景 |
| 悲观锁 | 认为冲突多，每次先加锁再操作 | 写多场景（MySQL 行锁） |
| 乐观锁 | 认为冲突少，先操作，提交时验证（CAS） | 读多写少（版本号机制） |

## 管程（Monitor）

> **一句话理解：** 管程 = 将共享资源和同步逻辑封装在一起，自动保证互斥访问。

**核心结论（可背）：**
```
管程的三个核心能力：
  ① 互斥访问：同一时刻只有一个进程能执行管程内的函数
  ② 条件同步：通过条件变量（wait/signal）协调执行顺序
  ③ 封装：隐藏同步细节，提供简洁接口

管程 vs 信号量：
  信号量：底层原语，灵活但容易误用（PV 顺序出错 → 死锁）
  管程：高级抽象，封装同步逻辑，更安全易用
  Java synchronized 块就是管程机制的实现
```

## 面试高频考点汇总

| 考点 | 核心答案 |
|---|---|
| 互斥 vs 同步的区别？ | 互斥：防止同时访问；同步：按顺序协调执行 |
| 死锁四个必要条件？ | 互斥、请求与保持、不剥夺、循环等待——破坏任一可预防 |
| 银行家算法的核心思想？ | 每次分配前检查是否存在安全序列，只有安全才分配 |
| PV 操作的含义？ | P = 申请资源（S--，可能阻塞）；V = 释放资源（S++，可能唤醒） |
| 生产者-消费者信号量设置？ | mutex=1（互斥）、empty=n（同步）、full=0（同步），P(mutex) 必须在 P(empty/full) 之后 |
| 哲学家进餐死锁如何解？ | 破坏循环等待：限制同时进餐人数；或两根都可用才拿；或奇偶顺序不同 |
| 自旋锁 vs 互斥锁？ | 自旋锁忙等不让出 CPU（适合短临界区）；互斥锁阻塞让出 CPU（适合长临界区） |
| 管程的作用？ | 封装共享资源和同步逻辑，自动保证互斥，简化并发编程 |
