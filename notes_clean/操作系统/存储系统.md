# 存储系统

---

## 速览

- 虚拟内存 = 以磁盘为后备，让进程看到比实际物理内存更大的连续地址空间。
- 虚拟地址 → 物理地址：通过页表映射；TLB 是页表的硬件缓存，加速地址转换。
- 页面置换算法：OPT（理想基准）→ LRU（性能最好）→ 时钟算法（LRU 近似，实际使用）→ FIFO（有 Belady 异常）。
- 分页（固定大小，无外部碎片）vs 分段（可变大小，符合逻辑，有外部碎片）。
- 缓冲区溢出攻击：写超界数据覆盖栈，防御：ASLR + DEP + 栈保护。
- 内存保护：用户态进程不能访问内核空间或其他进程空间，通过页表权限位实现。

## 虚拟内存

> **一句话理解：** 虚拟内存让每个进程以为自己独占整个地址空间，实际上由操作系统和硬件协同管理物理内存和磁盘。

**核心结论（可背）：**
```
虚拟内存三大能力：
  ① 高速缓存：将磁盘上的数据按需装入内存（缺页时才加载）
  ② 内存管理：每个进程独立的虚拟地址空间，OS 管理物理内存分配
  ③ 内存保护：不同进程间隔离，通过页表权限位防止越权访问

虚拟页的三种状态：
  未分配（Free）   → 进程未申请，不占物理内存和磁盘
  未缓存（Cached on Disk）→ 已分配但在磁盘上（不在内存中）
  已缓存（In Memory） → 已加载到物理内存中的页
```

**地址转换机制：**
```mermaid
flowchart LR
    VA["虚拟地址\n(Virtual Address)"] -->|"MMU 查页表"| TLB["TLB\n(Translation Lookaside Buffer)"]
    TLB -->|"命中（Hit）"| PA["物理地址\n(Physical Address)"]
    TLB -->|"未命中（Miss）\n查页表"| PT["页表\n(Page Table)"]
    PT --> PA
    PT -->|"页不在内存\n缺页中断"| Disk["磁盘\n(Page Fault)"]
    Disk -->|"调入内存"| PA
```

**TLB（Translation Lookaside Buffer）：**
```
TLB = 页表的硬件缓存（类似 CPU 的 L1/L2 缓存）
  命中：直接从 TLB 获取物理页号，无需访问内存中的页表（快）
  未命中：访问内存中的页表，将结果写入 TLB，替换最旧的条目
  上下文切换时 TLB 被清空（或打上进程标记 ASID 避免清空开销）
```

**多级页表：**
```
问题：32 位地址空间 + 4KB 页 = 100 万个页表项 × 4 字节 = 4MB 页表，每个进程都要占用
解决：多级页表（二级/三级），用时间换空间
  只有实际使用的虚拟地址才会分配对应的页表项
  64 位系统通常用 4 级页表（如 x86-64）
```

**面试官常问：**
- 为什么需要虚拟内存？→ 进程独立地址空间（隔离安全）、充分利用物理内存（按需加载）、支持比物理内存更大的程序运行。

## 页面置换算法

> **一句话理解：** 内存满了要换出某页，选哪个页换出决定了缺页率——换出"未来最久不用"的页最优。

**核心结论（可背）：**
| 算法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| OPT（最优置换） | 换出未来最长时间不用的页 | 理论最低缺页率（基准对比用） | 无法实现（需预知未来） |
| FIFO（先进先出） | 换出最早进入内存的页 | 实现简单 | 性能差；有 Belady 异常（帧越多缺页反而越多） |
| LRU（最近最少使用） | 换出最久未被访问的页 | 性能接近 OPT，无 Belady 异常 | 实现开销大（每次访问需更新记录） |
| 时钟算法（Clock/Second-Chance） | LRU 近似：访问位=1 给第二次机会；=0 则换出 | 开销远小于 LRU，性能好于 FIFO | 近似算法，不完全准确 |
| 增强型时钟（NRU） | 同时考虑访问位和修改位，优先换出未修改的脏页 | 减少写回磁盘次数 | 实现更复杂 |

**时钟算法详解：**
```
所有物理帧排成环形链表，时针指向一个帧

需要换出时：
  检查当前帧：
    访问位 = 1 → 给第二次机会（清零访问位），时针往前
    访问位 = 0 → 选中换出，替换为新页，时针往前

增强型（NRU）四类优先级（从好到差的换出顺序）：
  (访问位=0, 修改位=0) → 最佳选择
  (访问位=0, 修改位=1) → 需要写回磁盘，稍差
  (访问位=1, 修改位=0) → 最近还用过
  (访问位=1, 修改位=1) → 最差选择
```

**Belady 异常：**
```
FIFO 特有的反常现象：分配更多物理帧，缺页率反而增加
LRU 和 OPT 不存在 Belady 异常
```

## 分页 vs 分段

> **一句话理解：** 分页面向操作系统（物理单位，固定大小），分段面向用户（逻辑单位，可变大小）。

**核心结论（可背）：**
| 维度 | 分页（Paging） | 分段（Segmentation） |
|---|---|---|
| 划分依据 | 物理，固定大小（如 4KB） | 逻辑，可变大小（代码段/数据段） |
| 外部碎片 | ❌ 无外部碎片（固定大小分配） | ✅ 有外部碎片（可变大小分配） |
| 内部碎片 | ✅ 有（最后一页可能未填满） | ❌ 无内部碎片 |
| 用户可见 | 用户不感知（透明） | 用户可感知（程序员按段组织代码） |
| 共享 | 按页共享（如共享代码页） | 按段共享（如共享代码段更自然） |
| 保护 | 通过页表权限位 | 通过段描述符权限位 |

**请求分页（按需分页）：**
```
程序运行时不全部装入内存，只装入当前需要的页
  缺页中断处理流程：
  ① 访问虚拟地址，查页表，有效位=0 → 触发缺页中断
  ② OS 在磁盘找到对应页面
  ③ 在内存找一个空闲帧（无空闲则用置换算法选牺牲页）
  ④ 若牺牲页被修改（脏页）→ 先写回磁盘
  ⑤ 将所需页从磁盘加载到选好的帧，更新页表有效位=1
  ⑥ 重新执行被中断的指令
```

## 内存保护机制

> **一句话理解：** 操作系统通过页表权限位，阻止进程访问不属于自己的内存区域。

**核心结论（可背）：**
```
内存保护的三层防线：
  ① 用户态 vs 内核态：用户程序不能直接访问内核地址空间
  ② 进程间隔离：每个进程有独立页表，进程 A 的虚拟地址不映射到进程 B 的物理内存
  ③ 页面级权限：页表项中有读/写/执行权限位，违反则触发保护故障

实现机制：
  页表中每个 PTE（Page Table Entry）包含权限位：
    R（可读）、W（可写）、X（可执行）
  MMU 在地址转换时同时检查权限，违反则触发段错误（Segmentation Fault）
```

## 缓冲区溢出攻击

> **一句话理解：** 写入超过缓冲区大小的数据，覆盖返回地址，攻击者控制程序跳转到恶意代码。

**核心结论（可背）：**
```
攻击原理：
  局部数组 buf[10] 在栈上，写入超过 10 字节的数据
  → 覆盖栈上的返回地址（Return Address）
  → 函数返回时跳转到攻击者注入的恶意代码

危害：
  权限提升（执行 root 级别命令）
  数据泄露、服务崩溃（DoS）
  植入后门（僵尸网络）
```

**防御措施：**
| 层次 | 防御方法 | 说明 |
|---|---|---|
| 编程层 | 使用安全函数（strncpy 代替 strcpy） | 函数本身带长度限制 |
| 编译层 | 栈保护（-fstack-protector） | 在返回地址前插入 canary 值，返回时校验 |
| OS 层 | DEP（数据执行保护） | 数据页面标记为不可执行，注入的代码无法执行 |
| OS 层 | ASLR（地址空间随机化） | 每次运行基址随机，攻击者无法预知目标地址 |

## 面试高频考点汇总

| 考点 | 核心答案 |
|---|---|
| 虚拟内存的三大能力？ | 高速缓存（按需装入）、内存管理（独立地址空间）、内存保护（页表权限） |
| TLB 的作用？ | 页表的硬件缓存，命中则直接获取物理地址，避免访问内存中的页表（加速地址转换） |
| 为什么 FIFO 有 Belady 异常？ | 只考虑进入时间，不考虑使用频率；更多帧可能让更长期的老页继续留在内存 |
| LRU 实现为什么困难？ | 需要每次内存访问时更新时间戳，硬件支持不足时开销极大 |
| 时钟算法原理？ | 环形链表+访问位；访问位=1 给第二次机会清零；访问位=0 换出 |
| 分页 vs 分段的区别？ | 分页：物理固定大小，无外部碎片；分段：逻辑可变大小，有外部碎片 |
| 缓冲区溢出如何防御？ | ASLR（随机基址）+ DEP（数据不可执行）+ 栈保护（canary 值） |
