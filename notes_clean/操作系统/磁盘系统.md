# 磁盘系统

---

## 速览

- SPOOLING = 用磁盘缓冲队列解耦 CPU 和慢速外设（如打印机），实现 I/O 与 CPU 并行。
- 磁盘调度目标：减少磁头寻道时间，提高 I/O 吞吐量，避免饥饿。
- 五种磁盘调度算法：FCFS、SSTF（最短寻道优先）、SCAN（电梯）、C-SCAN（循环扫描）、LOOK/C-LOOK。
- 三种 I/O 控制方式：程序控制（轮询，CPU 忙等）、中断驱动（CPU 发命令后干其他事）、DMA（批量数据不经 CPU 直接传输）。
- SSD 没有机械磁头，磁盘调度算法对 SSD 意义不大。

---

## SPOOLING 技术

> **一句话理解：** SPOOLING = 用磁盘做"缓存队列"，把独占的慢速外设虚拟化为可多人使用的共享设备。

**核心结论（可背）：**
```
SPOOLING（联机外围同时操作）解决的问题：
  CPU 速度 >> 打印机/磁带等慢速外设的速度 → CPU 大量等待浪费

解决方式：
  用户提交任务 → 存入磁盘缓冲队列（输入井/输出井）→ 由后台进程异步处理
  CPU 从磁盘读取，不直接等待慢速外设
```

**四组件模型：**
| 组件 | 作用 |
|---|---|
| 输入井/输出井 | 磁盘上的队列，存放待处理输入和待输出数据 |
| 输入/输出缓冲区 | 内存中的中转区，减少频繁访问磁盘的开销 |
| 输入/输出进程 | 后台守护进程，负责在设备和磁盘之间搬运数据 |
| 井管理程序 | 任务调度核心，控制队列中任务的执行顺序 |

**三大特性：**
```
① 缓解 CPU 和 I/O 速度不匹配矛盾（磁盘比外设快得多）
② 将独占设备改造为虚拟共享设备（多个进程可"同时"使用同一台打印机）
③ 一个物理设备 → 多个逻辑设备（每个进程看到独立的逻辑打印机）
```

**现代应用：**
```
打印机队列：Windows 打印后台服务（Print Spooler）
邮件服务器：SMTP 将待发邮件存入发送队列
消息队列：Kafka、RabbitMQ 借鉴了 SPOOLING 的设计思想
```

**易错点：**
- ❌ SPOOLING 只能用于打印机 → SPOOLING 思想广泛用于所有慢速外设和批处理系统。
- ❌ SPOOLING 和 Buffering 是同一回事 → Buffering 是内存中转区，SPOOLING 用磁盘做大容量缓存，两者是配合关系。

## 磁盘调度算法

> **一句话理解：** 磁盘的性能瓶颈在寻道时间（磁头移动），调度算法就是优化磁头移动顺序，减少总移动距离。

**核心结论（可背）：**
| 算法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| FCFS（先来先服务） | 按请求到达顺序处理 | 简单公平 | 磁头大量往返，性能差 |
| SSTF（最短寻道时间优先） | 优先处理离磁头最近的请求 | 平均寻道时间短 | 远端请求饥饿 |
| SCAN（电梯算法） | 磁头单向移动，到头后反向，沿途处理请求 | 避免饥饿，比 FCFS 高效 | 边缘请求等待时间较长 |
| C-SCAN（循环扫描） | 只往一个方向移动，到头后直接跳回起点 | 请求响应时间更均匀 | 回程不处理请求，有额外开销 |
| LOOK | 类似 SCAN，但不走到磁盘边界，只到最后一个请求位置 | 减少不必要移动 | 实现略复杂 |
| C-LOOK | 类似 C-SCAN，但到最后一个请求位置后直接跳回 | 最高效，响应时间均匀 | 实现复杂 |

**磁头移动示意（SCAN 电梯算法）：**
```
磁道号：0 ←────────────────────────────────→ 200
当前磁头位置：53
请求队列：98, 183, 37, 122, 14, 124, 65, 67

SCAN（向右）：53 → 65 → 67 → 98 → 122 → 124 → 183 → 200（端点）→ 37 → 14
LOOK（向右）：53 → 65 → 67 → 98 → 122 → 124 → 183（最后）→ 37 → 14
```

**选择依据：**
```
FCFS   → 低负载或公平性要求高
SSTF   → 响应速度优先，可接受少数请求等待较久
SCAN/LOOK → 通用场景，负载较高，避免饥饿
C-SCAN/C-LOOK → 高负载，需要均匀响应时间（推荐）
```

**面试官常问：**
- SSD 为什么不需要磁盘调度算法？→ SSD 没有机械磁头，任意地址访问时间一致，不存在寻道时间。

## I/O 控制方式

> **一句话理解：** 三种方式本质是让 CPU 参与 I/O 的程度越来越低：轮询 → 中断 → DMA 完全不需要 CPU。

**核心结论（可背）：**
| 控制方式 | CPU 参与程度 | 原理 | 适用场景 |
|---|---|---|---|
| 程序控制 I/O（轮询） | 最高，CPU 忙等 | CPU 不断查询设备状态寄存器，直到设备就绪 | 简单低速设备，早期系统 |
| 中断驱动 I/O | 中等，CPU 处理中断 | CPU 发命令后继续执行其他任务；设备就绪后发中断通知 CPU | 键盘、鼠标等中速设备 |
| DMA（直接存储器访问） | 最低，仅启动/结束 | DMA 控制器直接负责数据块在内存和设备间的传输，不经 CPU | 磁盘、网卡等高速大数据量设备 |

**DMA 工作流程：**
```
① CPU 向 DMA 控制器发送：源地址、目标地址、数据块大小
② DMA 控制器自行完成整块数据传输（不占用 CPU）
③ 传输完成，DMA 控制器发中断通知 CPU
④ CPU 处理中断（仅做收尾工作）

优点：CPU 只在开始和结束时介入，大幅提升批量数据传输效率
缺点：需要额外硬件（DMA 控制器），成本较高
```

## 面试高频考点汇总

| 考点 | 核心答案 |
|---|---|
| SPOOLING 解决了什么问题？ | CPU 与慢速外设速度不匹配；把独占设备虚拟化为共享设备 |
| SCAN 和 C-SCAN 的区别？ | SCAN 双向移动，C-SCAN 单向+跳回起点；C-SCAN 响应时间更均匀 |
| LOOK 相比 SCAN 的优化？ | LOOK 不移动到磁盘边界，只到最后一个请求位置，减少不必要的磁头移动 |
| SSTF 的缺点？ | 离磁头远的请求可能长期得不到服务（饥饿现象） |
| DMA 的优点？ | CPU 只负责启动和结束，大块数据传输不占用 CPU，效率最高 |
| 为什么 SSD 不需要磁盘调度？ | SSD 没有机械磁头，无寻道时间，任意位置访问时间相同 |
