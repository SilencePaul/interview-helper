# 单例模式

---

## 速览

- 单例模式 = 确保全局只有一个实例，提供统一访问点。
- 六种实现方式，面试重点：**双重检查锁（DCL）** 和 **静态内部类**。
- DCL 必须加 `volatile`，原因是防止指令重排导致返回"半初始化"实例。
- 枚举是最安全的单例，天然防反射和序列化破坏。
- Spring 的 Bean 默认就是单例（scope=singleton）。

---

## 六种实现方式对比

> **一句话理解：** 懒汉延迟初始化，饿汉提前初始化，双检锁两者兼顾，枚举最安全。

**核心结论（可背）：**
| 实现方式 | 线程安全 | 懒加载 | 防反射 | 防序列化 | 推荐度 |
|---|---|---|---|---|---|
| 懒汉式（无锁） | ❌ | ✅ | ❌ | ❌ | ❌ 不用 |
| 饿汉式 | ✅ | ❌ | ❌ | ❌ | ✅ 简单场景 |
| 懒汉式（方法加锁） | ✅ | ✅ | ❌ | ❌ | ❌ 性能差 |
| **双重检查锁（DCL）** | ✅ | ✅ | ❌ | ❌ | ✅✅ 推荐 |
| **静态内部类** | ✅ | ✅ | ❌ | ❌ | ✅✅ 推荐 |
| **枚举** | ✅ | ❌ | ✅ | ✅ | ✅✅ 最安全 |

---

## 懒汉式（线程不安全）

```java
// ❌ 多线程下可能创建多个实例
class Singleton {
    private static Singleton instance;
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();  // 多线程同时进入 → 多个实例
        }
        return instance;
    }
}
```

---

## 饿汉式（线程安全，无懒加载）

```java
// ✅ 类加载时就创建，简单安全
class Singleton {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() {}
    public static Singleton getInstance() { return INSTANCE; }
}
```
- 缺点：类加载即创建，若从未使用则浪费内存。

---

## 双重检查锁（DCL）— 推荐

> **一句话理解：** 第一次检查避免每次加锁，第二次检查避免重复创建，`volatile` 禁止指令重排。

```java
class DCLSingleton {
    // volatile 禁止指令重排，避免返回"半初始化"实例
    private static volatile DCLSingleton instance;
    private DCLSingleton() {}

    public static DCLSingleton getInstance() {
        if (instance == null) {               // 第一次检查：避免每次加锁
            synchronized (DCLSingleton.class) {
                if (instance == null) {       // 第二次检查：防止重复创建
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```

**为什么必须加 `volatile`（必考）：**
```
instance = new DCLSingleton() 不是原子操作，分三步：
  ① 分配内存
  ② 初始化对象
  ③ 将 instance 指向内存地址

JVM 指令重排可能变成：① → ③ → ②
此时另一线程看到 instance != null，但对象未初始化完毕 → 拿到半成品实例 ❌

volatile 禁止指令重排，保证 ③ 在 ② 之后执行 ✅
```

---

## 静态内部类 — 推荐

> **一句话理解：** 利用 JVM 类加载机制，外部类加载时静态内部类不加载，调用时才初始化，天然懒加载+线程安全。

```java
class Singleton {
    private Singleton() {}

    // 静态内部类：只有调用 getInstance() 时才加载，JVM 保证线程安全
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```
- 懒加载：Holder 类只在调用 `getInstance()` 时才被加载。
- 线程安全：JVM 类初始化加锁，保证 `INSTANCE` 只创建一次。
- 无需 `synchronized`，性能好。

---

## 枚举 — 最安全

> **一句话理解：** JVM 天然保证枚举单例，防反射防序列化，但不能懒加载。

```java
public enum Singleton {
    INSTANCE;

    public void doSomething() { /* 业务方法 */ }
}

// 使用
Singleton.INSTANCE.doSomething();
```

**枚举为什么最安全：**
- **防反射**：JVM 禁止通过反射调用枚举构造器，强行调用会抛 `IllegalArgumentException`。
- **防序列化**：枚举的序列化/反序列化由 JVM 特殊处理，不会创建新实例。

---

## 反射和序列化破坏单例

**核心结论（可背）：**
| 攻击方式 | 问题 | 解决方案 |
|---|---|---|
| 反射 | `Constructor.setAccessible(true)` 强制调用私有构造器 | 构造器中判断 instance != null 则抛异常；或使用枚举 |
| 序列化 | 反序列化会调用构造器创建新实例 | 重写 `readResolve()` 返回现有实例；或使用枚举 |

---

## 如何选择实现方式

**核心结论（可背）：**
```
需要懒加载？
  ├── 是 → 需要传参初始化？
  │         ├── 是 → 双重检查锁（DCL）
  │         └── 否 → 静态内部类（更简洁）
  └── 否 → 需要防反射/序列化？
              ├── 是 → 枚举
              └── 否 → 饿汉式（最简单）
```

## 使用场景

| 场景 | 说明 |
|---|---|
| 数据库连接池 | 全局唯一，统一管理连接 |
| 线程池 | 全局唯一，避免线程资源浪费 |
| 全局配置管理 | 所有模块读取同一份配置 |
| 日志管理器 | 统一写入，避免日志混乱 |
| Spring Bean | 默认 scope=singleton，IoC 容器管理 |

## 面试高频考点汇总

| 考点 | 核心答案 |
|---|---|
| DCL 为什么需要 volatile？ | 防止指令重排，避免返回半初始化实例 |
| 静态内部类为什么线程安全？ | JVM 类初始化加锁，保证只执行一次 |
| 枚举为什么最安全？ | JVM 级别防反射和序列化破坏 |
| 如何选择实现方式？ | 不需懒加载用饿汉/枚举；需懒加载用静态内部类；需传参用 DCL |
| Spring 中单例是如何实现的？ | IoC 容器缓存 Bean 实例，getBean 时返回同一对象 |
| 懒汉式为什么线程不安全？ | 多线程同时通过 `if(null)` 检查，各自创建实例 |
