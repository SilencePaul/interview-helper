# 单例模式

---

## 速览

- 单例模式 = 确保全局只有一个实例，提供统一访问点。
- 六种实现方式，面试重点：**双重检查锁（DCL）** 和 **静态内部类**。
- DCL 必须加 `volatile`，原因是防止指令重排导致返回"半初始化"实例。
- 枚举是最安全的单例，天然防反射和序列化破坏。
- Spring 的 Bean 默认就是单例（scope=singleton）。

---

## 六种实现方式对比

> **一句话理解：** 懒汉延迟初始化，饿汉提前初始化，双检锁两者兼顾，枚举最安全。

**核心结论（可背）：**
| 实现方式 | 线程安全 | 懒加载 | 防反射 | 防序列化 | 推荐度 |
|---|---|---|---|---|---|
| 懒汉式（无锁） | ❌ | ✅ | ❌ | ❌ | ❌ 不用 |
| 饿汉式 | ✅ | ❌ | ❌ | ❌ | ✅ 简单场景 |
| 懒汉式（方法加锁） | ✅ | ✅ | ❌ | ❌ | ❌ 性能差 |
| **双重检查锁（DCL）** | ✅ | ✅ | ❌ | ❌ | ✅✅ 推荐 |
| **静态内部类** | ✅ | ✅ | ❌ | ❌ | ✅✅ 推荐 |
| **枚举** | ✅ | ❌ | ✅ | ✅ | ✅✅ 最安全 |

- 分布式场景下单例失效（多进程/多 JVM），需用 Redis 等外部锁保证全局唯一

## 双重检查锁（DCL）— 推荐

> **一句话理解：** 第一次检查避免每次加锁，第二次检查避免重复创建，`volatile` 禁止指令重排。

**核心结论（可背）：**
```java
class DCLSingleton {
    // volatile 禁止指令重排，避免返回"半初始化"实例
    private static volatile DCLSingleton instance;
    private DCLSingleton() {}

    public static DCLSingleton getInstance() {
        if (instance == null) {               // 第一次检查：避免每次加锁
            synchronized (DCLSingleton.class) {
                if (instance == null) {       // 第二次检查：防止重复创建
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```

**为什么必须加 `volatile`（必考）：**
```
instance = new DCLSingleton() 不是原子操作，分三步：
  ① 分配内存
  ② 初始化对象
  ③ 将 instance 指向内存地址

JVM 指令重排可能变成：① → ③ → ②
此时另一线程看到 instance != null，但对象未初始化完毕 → 拿到半成品实例 ❌

volatile 禁止指令重排，保证 ③ 在 ② 之后执行 ✅
```

🎯 **Interview Triggers:**
- DCL 的两次 null 检查各自解决什么问题？（MECHANISM）
- 不加 volatile 会有什么问题，举例说明？（FAILURE）
- synchronized 锁的是什么？能用实例锁吗？（DETAIL）

🧠 **Question Type:** mechanism · failure case · concurrency detail

🔥 **Follow-up Paths:**
- 指令重排 → JVM 优化 → volatile 禁止重排原理
- 第一次检查 → 避免锁竞争（性能优化）
- 第二次检查 → 锁内再检查 → 防多线程竞态条件

🛠 **Engineering Hooks:**
- DCL 是高并发初始化场景的标准写法（懒加载 + 线程安全 + 高性能）
- Java 5+ 的 JMM 修订后 volatile 才真正解决 DCL 问题（旧版 JVM 仍有风险）
- 实际项目中推荐静态内部类代替 DCL，代码更简洁且无 volatile 争议

## 静态内部类 — 推荐

> **一句话理解：** 利用 JVM 类加载机制，外部类加载时静态内部类不加载，调用时才初始化，天然懒加载+线程安全。

**核心结论（可背）：**
```java
class Singleton {
    private Singleton() {}

    // 静态内部类：只有调用 getInstance() 时才加载，JVM 保证线程安全
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```
- 懒加载：Holder 类只在调用 `getInstance()` 时才被加载。
- 线程安全：JVM 类初始化加锁，保证 `INSTANCE` 只创建一次。
- 无需 `synchronized`，性能好。

🎯 **Interview Triggers:**
- 静态内部类单例的线程安全由谁保证？（MECHANISM）
- 它和 DCL 的懒加载实现有什么本质区别？（COMPARISON）
- 这种方式能防反射攻击吗？（LIMITATION）

🧠 **Question Type:** mechanism · comparison · limitation

🔥 **Follow-up Paths:**
- JVM 类初始化 → clinit 加锁 → 保证只执行一次
- 静态内部类 vs DCL → 代码简洁 vs 支持传参
- 反射攻击 → Constructor.setAccessible → 枚举防御方案

🛠 **Engineering Hooks:**
- 静态内部类是实际项目中最常用的懒汉单例写法（简洁、安全、高效）
- 适用于无需传参初始化的场景（需要传参则用 DCL）
- 框架源码中大量使用此模式，如 Spring 部分内置组件

## 枚举 — 最安全

> **一句话理解：** JVM 天然保证枚举单例，防反射防序列化，但不能懒加载。

**核心结论（可背）：**
```java
public enum Singleton {
    INSTANCE;

    public void doSomething() { /* 业务方法 */ }
}

// 使用
Singleton.INSTANCE.doSomething();
```

**枚举为什么最安全：**
- **防反射**：JVM 禁止通过反射调用枚举构造器，强行调用会抛 `IllegalArgumentException`。
- **防序列化**：枚举的序列化/反序列化由 JVM 特殊处理，不会创建新实例。

🎯 **Interview Triggers:**
- 枚举单例是怎么防止反射攻击的？（MECHANISM）
- 为什么说枚举单例是最推荐的实现？（TRADEOFF）
- 枚举的缺点是什么，什么场景不能用？（LIMITATION）

🧠 **Question Type:** mechanism · recommendation · limitation

🔥 **Follow-up Paths:**
- JVM 枚举反序列化 → 按名字查找已有实例 → 不新建对象
- 枚举不支持懒加载 → 类加载即创建 → 内存浪费风险
- Android 平台不推荐枚举 → DEX 文件膨胀 → 改用常量

🛠 **Engineering Hooks:**
- Effective Java 作者 Joshua Bloch 力荐枚举单例（最简洁+最安全）
- 适用于全局唯一且不需要延迟加载的工具类（如序列化器、配置加载器）
- 分布式系统中枚举单例仅保证 JVM 内唯一，跨进程需额外设计

## 反射和序列化破坏单例

**核心结论（可背）：**
| 攻击方式 | 问题 | 解决方案 |
|---|---|---|
| 反射 | `Constructor.setAccessible(true)` 强制调用私有构造器 | 构造器中判断 instance != null 则抛异常；或使用枚举 |
| 序列化 | 反序列化会调用构造器创建新实例 | 重写 `readResolve()` 返回现有实例；或使用枚举 |

🎯 **Interview Triggers:**
- 反射如何破坏单例？怎么防御？（ATTACK & DEFENSE）
- 序列化为什么会破坏单例，readResolve 原理是什么？（MECHANISM）
- 如果面试官让你写一个完全安全的单例，你怎么写？（SYNTHESIS）

🧠 **Question Type:** attack & defense · mechanism · synthesis

🔥 **Follow-up Paths:**
- 反射攻击 → setAccessible → 构造器防御代码
- 序列化 → ObjectInputStream → readResolve 钩子
- 枚举 → JVM 保证 → 无需任何额外防御代码

🛠 **Engineering Hooks:**
- 序列化场景下单例必须实现 `readResolve()`，否则反序列化出新实例
- 安全敏感场景（如密钥管理器）必须考虑反射/序列化攻击防御
- 枚举单例是唯一无需额外防御即可抵御全部攻击的实现

## 面试高频考点汇总

| 考点 | 核心答案 |
|---|---|
| DCL 为什么需要 volatile？ | 防止指令重排，避免返回半初始化实例 |
| 静态内部类为什么线程安全？ | JVM 类初始化加锁，保证只执行一次 |
| 枚举为什么最安全？ | JVM 级别防反射和序列化破坏 |
| 如何选择实现方式？ | 不需懒加载用饿汉/枚举；需懒加载用静态内部类；需传参用 DCL |
| Spring 中单例是如何实现的？ | IoC 容器缓存 Bean 实例，getBean 时返回同一对象 |
| 懒汉式为什么线程不安全？ | 多线程同时通过 `if(null)` 检查，各自创建实例 |
