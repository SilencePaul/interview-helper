# 计算机网络基础

## 速览

- OSI 七层 vs TCP/IP 四层：前者是理论标准，后者是实际实现。
- IP 地址（网络层逻辑地址）vs MAC 地址（链路层物理地址）：ARP 做桥接。
- NAT 解决 IPv4 地址枯竭，让多设备共享一个公网 IP。
- CDN 通过就近缓存加速静态资源访问。

## OSI 七层 vs TCP/IP 四层

> **一句话理解：** OSI 是理论模型，TCP/IP 是实际互联网标准，两者结构相似但层数不同。

**核心结论（可背）：**
| TCP/IP 层 | 对应 OSI 层 | 主要协议 | 功能 |
|---|---|---|---|
| 应用层 | 应用层 + 表示层 + 会话层 | HTTP、HTTPS、DNS、FTP、SMTP | 用户接口、数据格式、会话管理 |
| 传输层 | 传输层 | TCP、UDP | 端到端传输，可靠性 |
| 网络层 | 网络层 | IP、ICMP、ARP、DHCP | 逻辑地址路由 |
| 链路层 | 数据链路层 + 物理层 | Ethernet、Wi-Fi | 物理传输，MAC 地址寻址 |

**OSI 七层（从上到下）：应用 → 表示 → 会话 → 传输 → 网络 → 数据链路 → 物理**

**数据封装过程（发送端）：**
```
应用层数据
    ↓ 加应用层头
传输层段（Segment）：TCP/UDP 头 + 数据
    ↓ 加传输层头
网络层包（Packet）：IP 头 + 段
    ↓ 加网络层头
数据链路层帧（Frame）：MAC 头 + 包
    ↓ 转为信号
物理层比特流传输

接收端反向剥离各层头部
```

🎯 **Interview Triggers:**
- OSI 七层和 TCP/IP 四层有什么区别，实际工程中用哪个？（COMPARISON）
- 数据从应用层发送到物理层，每一层都做了什么？（MECHANISM）
- 为什么要分层设计网络协议栈，分层有什么好处和代价？（WHY）
- 网络层和传输层分别解决什么问题，职责边界在哪里？（CONCEPT）

🧠 **Question Type:** 概念对比 · 分层机制解释

🔥 **Follow-up Paths:**
- OSI 七层 → 各层典型协议举例
- TCP/IP 四层 → 传输层 TCP vs UDP 差异
- 数据封装 → 抓包分析各层头部字段
- 网络层 → IP 路由选路机制
- 链路层 → 以太网帧结构与 MTU

🛠 **Engineering Hooks:**
- 排查网络问题时按层自下而上定位：先确认物理连通，再查 IP 可达，再查端口监听，再查应用协议。
- tcpdump/Wireshark 抓包时能直观看到各层封装，分析时先看 IP 层再看传输层再看应用层载荷。
- 容器网络（如 Docker bridge/overlay）本质上是在链路层和网络层之间加了虚拟层，理解分层模型有助于排查容器网络故障。
- MTU（最大传输单元）是链路层约束，超过 1500 字节时 IP 层会分片，影响性能，生产环境需注意 jumbo frame 配置。

## IP 地址 vs MAC 地址

> **一句话理解：** IP 是逻辑地址（可变，用于路由），MAC 是物理地址（固定，用于局域网寻址）。

**核心结论（可背）：**
| 维度 | IP 地址 | MAC 地址 |
|---|---|---|
| 所属层 | 网络层（OSI 第 3 层） | 链路层（OSI 第 2 层） |
| 作用范围 | 跨网段，全球路由 | 仅局域网内 |
| 格式 | IPv4：点分十进制（192.168.1.1）；IPv6：冒号分隔十六进制 | 16 进制（00:1A:2B:3C:4D:5E） |
| 是否可变 | 可变（DHCP 动态分配）| 硬件固定（可软件伪造） |
| 唯一性 | 同一网络内唯一 | 全球唯一 |

🎯 **Interview Triggers:**
- IP 地址和 MAC 地址分别解决什么问题，为什么两者缺一不可？（WHY）
- 路由器转发数据包时，IP 地址和 MAC 地址分别怎么变化？（MECHANISM）
- MAC 地址既然全球唯一，为什么不直接用 MAC 地址做路由？（COMPARISON）
- DHCP 动态分配 IP 地址的过程是怎样的？（MECHANISM）

🧠 **Question Type:** 概念区分 · 协议机制说明

🔥 **Follow-up Paths:**
- IP 地址 → 子网划分与 CIDR 表示法
- MAC 地址 → ARP 协议如何完成 IP 到 MAC 的映射
- IP 地址 → IPv4 耗尽问题与 NAT/IPv6 应对方案
- DHCP → 动态地址分配流程与租约机制
- 路由器转发 → 每跳更新 MAC 但保留 IP 的原理

🛠 **Engineering Hooks:**
- 云服务器弹性 IP 本质是静态 NAT：公网 IP 绑定到实例私有 IP，重启后公网 IP 不变但实例可能更换物理机（MAC 变了）。
- Kubernetes Pod IP 是虚拟 IP，跨节点通信依赖 CNI 插件在底层封装额外的 IP/MAC 头，排查 Pod 网络不通时需关注 CNI 层。
- 容器/VM 迁移后 ARP 缓存可能过期导致短暂不通，可用 arping 主动发送免费 ARP 刷新邻居表。
- 生产环境中网络设备的 MAC 地址可用于安全审计，交换机 MAC 地址表异常（如同一端口大量 MAC）是网络攻击的重要信号。

## ARP 协议

> **一句话理解：** ARP 把 IP 地址翻译成 MAC 地址，是网络层和链路层之间的桥梁。

**核心结论（可背）：**
```
场景：设备 A 知道目标 IP，但不知道 MAC 地址，无法发送以太网帧

ARP 工作流程：
① 设备 A 广播 ARP 请求："谁的 IP 是 192.168.1.2？告诉我你的 MAC"
② 目标设备 B 收到后单播回复："是我，MAC 是 XX:XX:XX:XX:XX:XX"
③ 设备 A 缓存 IP→MAC 映射（ARP 缓存），后续直接查缓存
```

**ARP 欺骗攻击：**
- 攻击者伪造 ARP 响应，将自己的 MAC 地址冒充网关 MAC。
- 受害者的流量全部发给攻击者 → 中间人攻击。
- 防御：静态 ARP 表、交换机动态 ARP 检测（DAI）。

**面试官常问：**
- ARP 局限性？→ 只能在局域网（广播域）内工作，不能跨子网。
- ARP 和 DNS 的区别？→ DNS 是域名→IP（应用层）；ARP 是 IP→MAC（链路层/网络层边界）。

🎯 **Interview Triggers:**
- ARP 协议的工作原理是什么，为什么需要广播？（MECHANISM）
- ARP 欺骗攻击的原理是什么，如何防御？（FAILURE）
- ARP 缓存的作用是什么，缓存过期了会发生什么？（CONCEPT）
- 跨子网通信时，ARP 是如何工作的，数据包怎么到达目标？（SCENARIO）
- 免费 ARP（Gratuitous ARP）是什么，有什么用途？（CONCEPT）

🧠 **Question Type:** 协议机制解释 · 安全攻防分析

🔥 **Follow-up Paths:**
- ARP 欺骗 → 中间人攻击（MITM）防御手段
- ARP 缓存 → 缓存老化时间与性能权衡
- 跨子网通信 → 网关/路由器的代理 ARP 机制
- ARP → IPv6 中的 NDP（邻居发现协议）替代方案
- 免费 ARP → 高可用场景（VIP 漂移）中的应用

🛠 **Engineering Hooks:**
- 服务器网卡故障切换（bonding/team）后需发送免费 ARP，通知局域网内其他设备更新 ARP 缓存，否则流量仍发往旧 MAC。
- 云环境中 VIP（虚拟 IP）漂移（如 keepalived 主备切换）依赖免费 ARP 快速通告新 MAC，切换时间通常在秒级。
- 交换机开启 DAI（Dynamic ARP Inspection）可验证 ARP 报文合法性，结合 DHCP Snooping 绑定表防御 ARP 欺骗。
- 线上排查网络不通时，`arp -n` 查看 ARP 缓存是基础步骤，条目为 incomplete 说明对端不可达或 ARP 响应丢失。

## NAT 网络地址转换

> **一句话理解：** NAT 让局域网内多台设备共享一个公网 IP，节省 IPv4 地址。

**核心结论（可背）：**
```
家庭网络示例：

内网设备 A (192.168.1.2:5000) → [路由器 NAT] → 公网 IP (203.0.113.1:8001) → 服务器
内网设备 B (192.168.1.3:5000) → [路由器 NAT] → 公网 IP (203.0.113.1:8002) → 服务器

路由器维护 NAT 表：
  内网 IP:端口  ←→  公网端口
  192.168.1.2:5000 ←→ :8001
  192.168.1.3:5000 ←→ :8002
```

**NAT 类型：**
| 类型 | 说明 |
|---|---|
| 静态 NAT | 一对一，固定内网 IP↔公网 IP |
| 动态 NAT | 多对多，从公网 IP 池中动态分配 |
| PAT（端口地址转换） | 多对一，通过端口区分不同内网设备，最常见 |

🎯 **Interview Triggers:**
- NAT 的工作原理是什么，路由器如何区分不同内网设备的回包？（MECHANISM）
- NAT 对点对点通信（如 P2P）有什么影响，如何穿透？（TRADEOFF）
- NAT 解决了 IPv4 地址不足的问题，它带来了哪些新问题？（TRADEOFF）
- 静态 NAT、动态 NAT、PAT 有什么区别，分别适用于什么场景？（COMPARISON）

🧠 **Question Type:** 机制解释 · 场景权衡分析

🔥 **Follow-up Paths:**
- NAT 穿透 → STUN/TURN/ICE 协议在 WebRTC 中的应用
- PAT → 端口映射与端口耗尽风险
- NAT → IPv6 普及后 NAT 是否还有存在意义
- 云环境 NAT → 弹性 IP、NAT 网关与 SNAT/DNAT 区别
- NAT 状态表 → 连接跟踪（conntrack）机制

🛠 **Engineering Hooks:**
- 高并发服务器需注意 NAT 网关端口耗尽（SNAT 端口池耗尽），症状为新连接建立失败但存量连接正常，可通过增加 NAT 网关 EIP 或调大端口范围缓解。
- Kubernetes 中 Service 的 ClusterIP 本质是 iptables/ipvs 实现的 DNAT，流量进入 Pod 前会做目的地址转换，理解 NAT 有助于排查 Service 访问异常。
- 微服务间调用链路上有多层 NAT（容器 NAT → 节点 NAT → VPC NAT）时，日志中的源 IP 可能已被转换，需在各层记录原始 IP 才能追溯。
- 防火墙状态检测（Stateful Inspection）依赖连接跟踪表，与 NAT 表协同工作；大流量场景下 conntrack 表满会导致新连接被丢弃，需调整 nf_conntrack_max 参数。

## CDN 内容分发网络

> **一句话理解：** CDN 把静态资源缓存到离用户最近的节点，减少延迟和源站压力。

**核心结论（可背）：**
```
用户请求静态资源（图片/JS/CSS）
    ↓
DNS 解析 → CDN 智能 DNS → 返回最近节点 IP
    ↓
请求打到 CDN 边缘节点
    ↓ 有缓存？
    是 → 直接返回（快）
    否 → 回源（请求源服务器）→ 缓存到节点 → 返回
```

**CDN 的三大价值：**
1. **减少延迟**：用户从最近节点取资源，RTT 大幅降低。
2. **减轻源站压力**：命中缓存的请求不到源站。
3. **高可用**：节点多，单节点故障可自动切换。

🎯 **Interview Triggers:**
- CDN 的工作原理是什么，智能 DNS 如何选择最近节点？（MECHANISM）
- CDN 缓存命中率低时该如何排查和优化？（SCENARIO）
- CDN 如何处理动态内容，动态加速和静态加速有什么区别？（COMPARISON）
- CDN 节点缓存了旧版本资源，如何强制刷新？（IMPLEMENTATION）
- CDN 在 DDoS 防护中能起到什么作用？（SCENARIO）

🧠 **Question Type:** 架构机制解释 · 工程场景优化

🔥 **Follow-up Paths:**
- CDN 智能 DNS → Anycast 与 GeoDNS 的区别
- CDN 缓存 → Cache-Control / Expires 头部控制缓存策略
- CDN 回源 → 回源策略（主备源、权重源）与回源鉴权
- CDN 安全 → WAF 与 DDoS 防护集成
- CDN 与 HTTP/2 → 多路复用减少连接数的协同效果

🛠 **Engineering Hooks:**
- 静态资源文件名加内容哈希（如 main.abc123.js）可实现长缓存（Cache-Control: max-age=31536000），版本更新时文件名变化自动失效旧缓存，无需手动刷新 CDN。
- CDN 回源时需在源站配置回源 IP 白名单，防止源站直接暴露到公网；同时源站域名与 CDN 域名分离，避免用户绕过 CDN 直连源站。
- 全站加速（DCDN）通过在 CDN 边缘节点与源站之间建立专线或优化路由，解决动态请求跨国延迟高的问题，适用于 API 接口加速场景。
- CDN 日志分析可用于监控攻击流量（异常 IP、UA、请求路径），结合实时带宽数据判断是否遭受 CC 攻击，触发限速或封禁规则。

## 网络分层的意义

> **一句话理解：** 分层让每层只关注自己的职责，便于独立开发、测试、替换。

**核心结论（可背）：**
- 层与层之间通过接口通信，上层不关心下层如何实现。
- 修改某一层实现不影响其他层（如从 IPv4 升级到 IPv6，应用层无感知）。
- 标准化的分层使不同厂商设备能互联互通。

🎯 **Interview Triggers:**
- 网络协议为什么要分层设计，不分层会有什么问题？（WHY）
- 分层设计有哪些缺点或性能代价？（TRADEOFF）
- 如果让你设计一个新的网络协议栈，你会如何划分层次？（IMPLEMENTATION）
- IPv4 升级到 IPv6 时，为什么应用层代码不需要修改？（MECHANISM）

🧠 **Question Type:** 架构设计原则 · 工程权衡分析

🔥 **Follow-up Paths:**
- 分层设计 → 软件工程中的分层架构（MVC、微服务）类比
- 协议替换 → QUIC 协议将传输层功能移到应用层的设计取舍
- 标准化 → RFC 标准制定流程与互联网治理
- 跨层优化 → TCP 卸载（TOE）与 RDMA 绕过内核的实现
- 网络虚拟化 → SDN 将控制平面与数据平面分离的思路

🛠 **Engineering Hooks:**
- QUIC 协议将可靠传输逻辑从内核态（TCP）移到用户态（应用层），牺牲了分层纯粹性，换来了更快的迭代速度和更低的连接建立延迟（0-RTT）。
- RDMA（远程直接内存访问）绕过操作系统内核网络栈，直接在网卡和用户态内存间传输数据，大幅降低延迟，常用于高性能计算和分布式存储（如 Ceph、HDFS）。
- eBPF 技术允许在内核网络栈各层注入自定义逻辑（如 Cilium CNI），在不破坏分层结构的前提下实现高性能网络策略，是云原生网络的重要基础。
- 分层抽象在排查故障时是系统化定位的依据：从物理层（网线/光模块）→ 链路层（交换机/VLAN）→ 网络层（路由/ACL）→ 传输层（端口/防火墙）→ 应用层（服务配置）逐层排查。
