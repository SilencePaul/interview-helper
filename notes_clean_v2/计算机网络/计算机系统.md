## 加密体系基础

> **一句话理解：** 对称加密快但密钥传输不安全，非对称加密安全但慢，HTTPS 结合两者优点。

**核心结论（可背）：**
| 类型 | 代表算法 | 密钥特点 | 速度 | 场景 |
|---|---|---|---|---|
| 对称加密 | AES、DES、ChaCha20 | 加解密同一密钥 | 快 | 大量数据加密（TLS 通信阶段） |
| 非对称加密 | RSA、ECDHE | 公钥加密，私钥解密 | 慢 | 密钥交换、数字签名（TLS 握手阶段） |
| 摘要算法 | MD5、SHA-256 | 单向不可逆 | 快 | 数据完整性校验（数字指纹） |

**为什么 HTTPS 不全程用非对称加密？**
非对称加密计算复杂度高，加密速度慢，不适合加密大量数据；对称加密速度是非对称的数倍，适合持续通信。

🎯 **Interview Triggers:**
- 对称加密和非对称加密的区别是什么，HTTPS 为什么要同时使用两者？（COMPARISON）
- MD5 和 SHA-256 都是摘要算法，为什么现在不推荐使用 MD5？（TRADEOFF）
- 如果只用对称加密，密钥交换过程中存在什么安全风险？（FAILURE）
- 非对称加密中公钥加密私钥解密，和私钥签名公钥验签分别用在什么场景？（CONCEPT）
- AES 和 RSA 在性能上的差距大概是多少数量级？（MECHANISM）

🧠 **Question Type:** 概念辨析 · 安全机制 · 性能对比

🔥 **Follow-up Paths:**
- 加密体系 → TLS 握手过程（非对称协商对称密钥的完整流程）
- 非对称加密 → 数字签名原理（私钥签名 + 公钥验签）
- 摘要算法 → CA 数字证书结构（证书内容 + 摘要 + CA 私钥签名）
- 对称加密 → AES 工作模式（ECB/CBC/GCM 区别）
- MD5 弱点 → 哈希碰撞攻击与彩虹表破解原理

🛠 **Engineering Hooks:**
- 密码存储不能使用 MD5 或 SHA-256 直接哈希，应使用专为密码设计的慢速哈希算法（bcrypt/Argon2），加盐防彩虹表攻击。
- HTTPS 通信阶段使用的对称加密算法现代推荐 AES-GCM（提供认证加密，同时保证机密性和完整性），避免使用 AES-CBC（需单独处理完整性）。
- 服务间内部 API 调用可使用 HMAC-SHA256 对请求内容签名，防止请求被篡改，签名密钥通过安全渠道（如 Vault）分发。
- 生产环境私钥文件权限应设置为 600（仅 owner 可读），存储在 HSM（硬件安全模块）中可进一步防止私钥泄露。

## HTTPS 三大安全保障机制

> **一句话理解：** 保密性靠加密，完整性靠摘要，身份验证靠证书。

**核心结论（可背）：**
```
保密性（机密性）：
  混合加密 = 非对称（握手）+ 对称（通信）
  任何第三方截获报文，无法解密

完整性：
  摘要算法（如 SHA-256）对数据生成「数字指纹」
  接收方重新计算指纹，与附带指纹对比
  任何篡改都会导致指纹不匹配

身份验证：
  CA 数字证书：权威机构对服务器公钥的背书
  客户端验证证书有效性 → 确认服务器身份真实
```

🎯 **Interview Triggers:**
- HTTPS 是如何同时保证保密性、完整性和身份验证的？（MECHANISM）
- 如果没有 CA 证书机制，HTTPS 会面临什么攻击风险？（FAILURE）
- 中间人攻击（MITM）是什么原理，HTTPS 是如何防御的？（FAILURE）
- 数字证书包含哪些信息，客户端如何验证证书的合法性？（CONCEPT）
- 自签名证书和 CA 颁发证书有什么区别，分别适用什么场景？（COMPARISON）

🧠 **Question Type:** 安全机制 · 攻击与防御 · 证书体系

🔥 **Follow-up Paths:**
- HTTPS 三大保障 → TLS 握手详细流程（证书验证在握手的哪个阶段）
- CA 证书 → 证书链验证（根证书 → 中间证书 → 服务器证书）
- 中间人攻击 → HSTS（HTTP Strict Transport Security）强制 HTTPS
- 证书透明度 → CT Log 机制防止 CA 误签证书
- 证书吊销 → CRL 和 OCSP 两种吊销查询机制对比

🛠 **Engineering Hooks:**
- 使用 Let's Encrypt 可免费申请受信任的 TLS 证书，配合 Certbot 自动续期（证书有效期 90 天），生产环境常见方案。
- 开发环境使用 `mkcert` 工具生成本地受信任的自签名证书，避免浏览器安全警告，比直接用自签名证书更方便。
- 生产环境应配置 HSTS 响应头（`Strict-Transport-Security: max-age=31536000; includeSubDomains`），防止 SSL Stripping 攻击降级到 HTTP。
- 监控证书过期时间，至少提前 30 天告警，避免证书到期导致服务不可访问（已有多起大厂因证书过期引发故障）。

## HTTP 版本演进

> **一句话理解：** 每一版都在解决上一版的性能或安全缺陷。

**核心结论（可背）：**
| 版本 | 核心特性 | 主要问题 |
|---|---|---|
| HTTP/1.0 | 短连接，每次请求新建 TCP | 连接开销大 |
| HTTP/1.1 | 持久连接（keep-alive），管道化 | 队头阻塞（HOL Blocking） |
| HTTP/2 | 多路复用（同一 TCP 多流并发），头部压缩 | TCP 层队头阻塞 |
| HTTP/3 | 基于 QUIC（UDP），内置 TLS 1.3 | 彻底解决队头阻塞 |

**队头阻塞（HOL Blocking）：**
```
HTTP/1.1 管道化：请求必须按顺序返回
  → 前一个响应慢，后面全部阻塞

HTTP/2 多路复用：TCP 层仍是顺序字节流
  → 一个 TCP 包丢失，整个连接所有流阻塞

HTTP/3（QUIC）：每个流独立，一个流丢包不影响其他流
  → 真正解决队头阻塞
```

🎯 **Interview Triggers:**
- 什么是 HTTP 队头阻塞，HTTP/1.1、HTTP/2、HTTP/3 各自是如何处理的？（MECHANISM）
- HTTP/2 的多路复用为什么仍然存在队头阻塞问题？（WHY）
- HTTP/3 基于 UDP 如何保证数据传输的可靠性？（IMPLEMENTATION）
- HTTP/1.1 的持久连接（keep-alive）解决了什么问题，还有什么局限？（TRADEOFF）
- 在移动网络弱网环境下，HTTP/3 相比 HTTP/2 有哪些具体优势？（SCENARIO）

🧠 **Question Type:** 版本演进 · 性能瓶颈分析 · 协议机制

🔥 **Follow-up Paths:**
- HTTP 版本演进 → QUIC 协议详细机制（流量控制/拥塞控制/连接迁移）
- HTTP/2 多路复用 → 二进制帧与 Stream ID 实现原理
- HTTP/3 → 0-RTT 握手原理与安全风险（重放攻击）
- 队头阻塞 → TCP 可靠传输机制（ACK/重传/滑动窗口）
- HTTP/2 头部压缩 → HPACK 静态表与动态表工作原理

🛠 **Engineering Hooks:**
- HTTP/2 在高丢包率网络（>2%）下性能可能不如 HTTP/1.1，因为单个 TCP 连接丢包会阻塞所有流；HTTP/3 在弱网下有明显优势。
- 开启 HTTP/2 后应移除之前为 HTTP/1.1 优化的域名分片策略（多 CDN 域名），HTTP/2 多路复用使域名分片反而增加握手开销。
- 服务端可通过 `Alt-Svc` 响应头告知客户端支持 HTTP/3，浏览器下次请求会尝试 QUIC 连接，实现渐进式升级。
- 抓包分析 HTTP/2 流量需使用 Wireshark 并配置 TLS 解密（导出浏览器 SSL key log），HTTP/2 的二进制帧格式无法直接阅读。

## TLS 1.2 vs TLS 1.3

**核心结论（可背）：**
| 维度 | TLS 1.2 | TLS 1.3 |
|---|---|---|
| 握手 RTT | 2-RTT | 1-RTT（0-RTT 重连） |
| 密钥交换 | RSA（不具备前向安全） | 强制 ECDHE（前向安全） |
| 支持的套件 | 多（含不安全的） | 精简，只保留安全套件 |

**前向安全性：**
即使服务器私钥泄露，也无法解密历史通信记录。TLS 1.3 的 ECDHE 每次握手生成新的临时密钥，实现前向安全。

🎯 **Interview Triggers:**
- TLS 1.3 相比 TLS 1.2 有哪些核心改进？（COMPARISON）
- 什么是前向安全性，为什么 TLS 1.2 的 RSA 密钥交换不具备前向安全？（CONCEPT）
- TLS 1.3 的 0-RTT 重连是如何实现的，存在什么安全风险？（TRADEOFF）
- TLS 握手的完整过程是怎样的，1-RTT 握手省去了哪些步骤？（MECHANISM）
- 如何在服务器上强制禁用 TLS 1.0/1.1，只允许 TLS 1.2 及以上版本？（IMPLEMENTATION）

🧠 **Question Type:** 协议对比 · 安全机制 · 握手流程

🔥 **Follow-up Paths:**
- TLS 1.3 → ECDHE 密钥交换算法原理（椭圆曲线 Diffie-Hellman）
- 前向安全 → 会话密钥派生机制（HKDF 密钥扩展函数）
- 0-RTT 重连 → PSK（Pre-Shared Key）会话恢复与重放攻击风险
- TLS 握手 → 证书验证流程（HTTPS 三大保障机制）
- TLS 版本 → PCI DSS 合规要求（禁用 TLS 1.0/1.1 的监管背景）

🛠 **Engineering Hooks:**
- Nginx 配置中通过 `ssl_protocols TLSv1.2 TLSv1.3` 明确指定支持的 TLS 版本，禁用不安全的 TLS 1.0/1.1，满足安全合规要求。
- TLS 1.3 的 0-RTT 模式存在重放攻击风险，对于幂等的 GET 请求可以开启，对于非幂等的写操作（如支付）应禁用 0-RTT。
- 使用 `openssl s_client -connect example.com:443 -tls1_3` 可验证服务器是否支持 TLS 1.3，生产环境部署后应验证。
- 推荐的 TLS 1.3 密码套件：`TLS_AES_256_GCM_SHA384` 和 `TLS_CHACHA20_POLY1305_SHA256`，后者在移动端（无 AES 硬件加速）性能更好。
- 可通过 SSL Labs（ssllabs.com/ssltest）对生产域名进行 TLS 配置安全评级，目标达到 A+ 评级。

## 面试高频考点汇总

| 考点 | 核心答案 |
|---|---|
| 对称 vs 非对称加密区别？ | 密钥数量（一个 vs 两个）；速度（快 vs 慢）；用途（通信 vs 握手） |
| HTTPS 三大保障？ | 保密性（混合加密）、完整性（摘要算法）、身份验证（CA 证书） |
| HTTP 各版本核心改进？ | 1.1 持久连接；2 多路复用；3 QUIC 消除队头阻塞 |
| 什么是队头阻塞？ | 前一个请求/数据包阻塞导致后续全部等待 |
| TLS 1.3 比 1.2 好在哪？ | 1-RTT 握手、强制前向安全（ECDHE）、精简套件 |
| 摘要算法的作用？ | 生成数据指纹，验证完整性，防篡改 |
